# Team MTG Agenda — SAFEWATCH Mode 認識合わせ

> 次回MTG用。デモを見せた後に議論を回すための進行台本。

## 次回MTGの目的

- SAFEWATCH Mode の**デモを全員で見て、完成イメージをそろえる**
- 必須機能・追加機能・やらないことの合意
- 役割分担とスケジュールの初版を決める
- 提出形式（URL共有 / PR）の確認方針を決める

## タイムボックス（目安 60分）

| 時間 | 内容 |
|---|---|
| 0-5分 | 目的共有・今日のゴール確認 |
| 5-15分 | SAFEWATCH デモ（`demo-script.md`） |
| 15-30分 | 議論①: 必須/追加/やらないこと |
| 30-45分 | 議論②: 役割分担・ボトルネック |
| 45-55分 | 議論③: スケジュール・提出形式 |
| 55-60分 | 決定事項の確認・宿題割当 |

## デモを見る観点（メンバーへ事前共有）

- これは「完成品」ではなく議論のたたき台であること
- モード切替時のメッセージ / 閾値超過のWARNING が「実用的に足りているか」
- 監視すべき項目は roll/pitch/altitude で十分か、他に要るか
- フライトコードだけで作るべきか、Lua を混ぜるべきか

## 議論したい論点

1. MVP の線引き（どこまでを卒業課題デモに含めるか）
2. 閾値の初期値と、パラメータ化の要否
3. 異常時アクション（警告のみ vs 自動処理）を今回やるか
4. 実装ボトルネック（`role-assignment-draft.md` の表）への対策
5. スケジュール（マイルストーンと締切）
6. 提出形式の確認担当

## メンバーから集めたいアイデア

- 監視したい追加項目（速度・バッテリー・GPS・ジオフェンス等）
- 「安全監視」ユースケースの具体シナリオ
- 発表で刺さりそうな見せ方
- 想定される質問・弱点（評価者目線）

## 最後に決めるべきこと（MTGの出口）

- [ ] MVP スコープの合意
- [ ] 役割分担（担当者名を埋める）
- [ ] 次マイルストーンの締切
- [ ] 提出形式の確認担当と期限
- [ ] 次回MTG日程
