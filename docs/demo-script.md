> 完成品ではなく「配管が通った叩き台」を見せ、**全体像・設計・過不足**を共有して、
> 各自の担当と次の一手を決めるためのデモ。所要 5〜8分。
> 方針の経緯は [design-decisions.md](design-decisions.md)、UI設計は [ux-flow.md](ux-flow.md) 参照。

## 製品ひとこと（最初に言う）

> **住所を入れるだけで、非エンジニアがその周辺を安全に巡回飛行できるツール。**

Mission Planner や QGroundControl は高機能だが非エンジニアには複雑。
うちは「**住所を入れるだけ ＋ 機体が安全を見守る**」に絞って差別化する（先行事例調査 → design-decisions D11）。

## システム構成（説明用・1枚で）

```
[ ブラウザのWebアプリ ]      ← 非エンジニアが操作（住所入力・ルート作成・実行）
   │  ルート/指示
   ▼  （裏でMAVLinkで接続）
[ バックエンド ] ── MAVLink ──▶ [ ArduPilot (SITL / 実機) ]
                                   └ フライトコードも最適化・拡張していく（③）
```

- **スマホアプリではなく、PCブラウザのWebアプリ**（地図を大きく使う。D16）
- 安全は「既存機能の統合」だけでなく、**ArduPilot のフライトコード側も自分たちで最適化・拡張する**と伝える（D18）

## 設計（3階建て） — 1枚で説明する図

```
        [ 住所 ]  例）名古屋城
           │
     ① 計画（Plan） ……… geocode.py：住所→座標／巡回ルート生成      担当: UI/UX(リーダー)
           ▼
   [ 座標 ＋ 巡回ルート ]
           │
     ② 実行（Execute）…… patrol_spine.py：ミッション化→送信→AUTO飛行  担当: 配管(?)
           ▼
   [ SITLで機体が飛ぶ ]
           │
     ③ 守る（Protect）…… ★まだ張りぼて（安全監視は未実装）           担当: フライトコード(?)
           ▼
      [ 安全に巡回 ]
```

- **魅せ場＝①**（非エンジニアの簡単さ）／**課題の芯＝③**（安全・フライトコード）
- ②はその2つを繋ぐ配管。今回のデモで"本物"なのは主に①→②。

## 過不足（正直マップ） — ここを堂々と見せるのがこのデモの肝

| 機能 | 状態 | メモ |
|---|---|---|
| 住所 → 座標 | ✅ 本物 | `geocode.py`（無料のNominatim） |
| 座標 → 四角ミッション → AUTO飛行 → 帰還 | ✅ 本物 | `patrol_spine.py`、SITLで完走確認済み |
| 1コマンド統合＋自動片付け | ✅ 本物 | `plan_and_fly.py "住所"` |
| 地図で動きが見える | ✅ 本物（ただし開発者用） | MAVProxyの地図。**専用UIではない** |
| 非エンジニア向けUI | ❌ 未実装 | 今はコマンドライン。画面設計は `ux-flow.md` にあり |
| 飛行中の安全監視（姿勢/範囲） | ⚠️ **張りぼて** | 画面の「姿勢OK・範囲内」は**固定文字。実際は何も見ていない** |
| 離陸前チェック（無茶な計画を拒否） | ❌ 未実装 | ← **この製品の独自価値。既存ツールに無い** |
| 地図で外周をなぞる / ルートのカスタム | ❌ 未実装 | 将来（design-decisions D8, D5） |

> 正直に「ここは張りぼて」「ここは未実装」と言うことが、認識合わせでは信頼になる。隠さない。

## 想定トラブル時の代替（フォールバック）

| 事象 | その場の対応 |
|---|---|
| 地図が動いて見えない | 先に飛ばし終えていた可能性。もう一度投入し「今から離陸」と実況する |
| MAVProxyが即終了 | ttyが無いのが原因。`script -qec ...` でpty付き起動（上記A） |
| ネット不通で住所変換失敗 | `geocode.py` の内蔵フォールバック地点（東京駅等）を使う |
| SITL起動が重い | 事前に起動しておく／録画で代替 |
