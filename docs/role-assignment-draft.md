## 前提: いまの実装状態（2026-07-04）

- ①計画〜③守るまで**一気通貫で動くデモが完成済み**（Webアプリ + SITL実飛行）
- 安全チェック11種（フェンス整合・空港・文化財・鉄道・送電線・学校病院・夜間・天気・目視距離ほか）は**実データで動作**
- ③守る層は Step1（設定の自動投入・フェンス自動フィット）+ Step2（Lua、SITL検証済）まで完了。**Step3（C++）が未着手**
- 残モック: 建物高さ・地形AGL（重いデータが必要）

→ つまり各担当の仕事は「ゼロから作る」ではなく「**動くものを深掘り・強化する**」。

## 各担当の「最初の一歩」（現コードベース準拠）

- **リーダー/①**: デモを一周流して詰まる箇所を洗い出し → UI改善タスク化。発表原稿（speaker-notes.md）を自分の言葉に直す
- **②配管**: `flight_service.py` を読む → 異常系（SITL起動失敗・接続断・Overpassタイムアウト）を意図的に起こして挙動確認 → 改善
- **③フライトコード**: `safety/patrol_safety.lua` をSITLで動かす（safety/README.md 手順）→ `AP_Arming_Copter.cpp` の `check_failed()` 群を読み、自作pre-arm（例: ジオフェンス未設定なら拒否）を1つ追加
- **検証**: README の手順で環境構築 → デモ完走 → 「電池を減らす」「フェンス外へ誘導」等の異常系を sitl-test-log.md の手順で再現
- **ドキュメント/発表**: speaker-notes.md（現行版）を読み、5分デモの時間を計って調整

## ボトルネック（リーダーが特に見る）

| リスク | 兆候 | 手当て |
|---|---|---|
| **Step3(C++)の難易度** | ビルド・mode構造の理解に時間がかかる | スコープは「pre-arm 1つ追加」に固定（D12）。Luaで代替可能なら差し替えも可 |
| 環境構築の個人差 | SITLが動かないメンバー | README + setup-progress.md で追える。動く人がペア対応 |
| 外部API依存 | Overpass混雑・Nominatim制限 | フォールバック実装済み。デモ本番はオンライン確認＋事前に一周流す |
| デモ本番の不確定要素 | 天気チェックが当日の実気象で⚠になる 等 | 「実データが動いている証拠」とポジティブに説明する台本にしておく（speaker-notes参照） |
| 統合の衝突 | 同じファイルを複数人が触る | 層で分担（①UI=app.html / ②=server・flight_service / ③=safety/）＋ team-process.md のブランチ運用 |
| 提出形式未確定 | URL共有かPRかで作業が変わる | 早期に講師確認（setup-progress.md の文面案） |
