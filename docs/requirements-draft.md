# Requirements Draft — 非エンジニア向け「住所で巡回」ドローン

> チームで確定する前の**たたき台**。MTGで追記・修正する前提。
> 方針の経緯は [design-decisions.md](design-decisions.md) を参照。

## 必須機能（MVP: これが無いとデモにならない）

| No | 層 | 機能 | 内容 | 備考 |
|---|---|---|---|---|
| M1 | ① | 住所→座標変換 | 住所/地名を緯度経度に変換（ジオコーディング） | 無料のNominatim等を想定 |
| M2 | ① | 巡回ルート自動生成 | 中心座標から既定サイズの外周（四角）ミッションを生成 | 既存 `mission_basic` の四角生成が土台 |
| M3 | ② | ミッション組み立て | 離陸→各ウェイポイント→帰還(RTL) の命令列を作る | dronekit `Command` で構築 |
| M4 | ② | アップロード＆実行 | SITLへ送信し **AUTOモード**で飛行開始 | arm→AUTO の一連 |
| M5 | ③ | 飛行中の安全監視 | 姿勢/高度/アーム状態を監視し、逸脱を検知 | **実装形は要設計**（下記・design-decisions D7/未解決参照） |
| M6 | 全 | SITLで一連の実演 | 住所入力→生成→飛行→監視→ログ確認 | デモ手順として再現可能に |

## 追加機能（余裕があれば / 次フェーズ = ロードマップ）

| No | 層 | 機能 | 内容 | 優先度 |
|---|---|---|---|---|
| A1 | ① | ルートのカスタム | サイズ/高度/周回数/形(四角・円)を数値で調整 | 中 |
| A2 | ① | 地図なぞりで外周指定 | 地図UI上でクリックして正確な外周を描く（Lv2） | 中 |
| A3 | ① | ルート保存・呼び出し | 「農地A」として登録→再実行（毎回同じルート） | 中 |
| A4 | ① | AI自然言語指示 | 「農地Aを高度15mで見回って」を解釈 | 低 |
| A5 | ③ | 異常時の自動アクション | 監視で異常→自動RTL/着陸など | 低 |
| A6 | ③ | 監視項目の追加 | 対地速度 / battery / GPS fix 等 | 低 |

## やらないこと（今回スコープ外）

| No | 項目 | 理由 |
|---|---|---|
| N1 | 実機飛行 | SITLで十分、安全・時間の都合 |
| N2 | スクショからのAI座標推定 | 座標情報を持たず精度不足＝安全リスク（design-decisions D3） |
| N3 | 地図UI・AI指示・ルートカスタム | まず「住所→自動巡回」を通す。将来機能へ |
| N4 | 異常時の自動アクション | 難易度が高い。今回は監視まで |
| N5 | 本家へのPull Request | 自リポジトリ内で完結・要確認事項 |

## 未確定事項（要確認）

- **③安全監視の実装形（最重要）**: AUTOで飛ぶ最中の監視を、
  (a)フライトコードの常時監視/フェイルセーフ / (b)独自モードで巡回＋監視 / (c)併走監視スクリプト のどれにするか。
- ジオコーディング提供元（Nominatim等）と利用条件
- SITL ホーム座標をジオコーディング結果に合わせる方法
- 既定の巡回サイズ・高度・周回数の初期値
- 提出形式（自分のrepo URL共有 / 指定repoへのPR）＝講師確認
- 本家へのPRは不要という理解でよいか＝講師確認

## チームで決めるべきこと

1. MVP の範囲（M1〜M6 のどこまでを発表デモに含めるか）
2. ③安全監視の実装形（a/b/c のどれか）
3. 既定の巡回パラメータ初期値
4. 役割分担（`role-assignment-draft.md` 参照 ※新方針での見直しが必要）
5. スケジュールとマイルストーン
6. 提出形式の確認担当と期限
