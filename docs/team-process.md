# チーム進行方法（team-process） — みまわり自作ドローン

> チームでの開発の回し方・守るルール・ドキュメントの地図。2026-07-04 作成。
> ここまでの1人開発で**実際に機能したやり方**を、チーム運用に翻訳したもの。

## 1. 開発の原則（ここまでの実績から）

1. **薄く一本通してから深掘る（walking skeleton, D14）**
   全部作り込んでから繋ぐのではなく、「住所→飛ぶ」の最小の背骨を先に通し、各ピースを順に本物へ差し替えてきた（張りぼてUI→実テレメトリ、サンプル鉄道→OSM実データ、固定135m→導出値）。新機能も同じ順で: **まず動く最小 → SITLで検証 → 深掘り**。

2. **決定は理由ごと記録する（design-decisions.md）**
   「何を・なぜ・何と比べて」をD番号で残す。決定は変えてよいが、**変えた理由を必ず残す**。議論が振り出しに戻るのを防ぐ、チーム最大の資産。

3. **実ソース・実データで裏取りしてから設計する**
   「独自モードで監視」が不可能なことは ArduPilot 実ソースを読んで判明した（D21）。RTL_ALT改名もソース確認で解決。**推測で設計せず、`~/GitHub/ardupilot` を grep する**。

4. **SITLで検証してから push**
   コミットには検証結果を書く。検証で見つかった不具合（GCS FS誤作動・dronekit同値タイムアウト等）は sitl-test-log.md に「発見と修正」として記録する。**未検証のものを「動く」と言わない**。

5. **正直表示の原則**
   モックは「モック」と明示、データ取得失敗は「確認できませんでした」と表示、サンプルデータは「デモ用データ」と書く。安全を扱う製品では、できていないことを隠すのが一番危険。

## 2. Git運用（軽量ルール）

- ブランチ: 現在 `master` 直コミット（1人開発）。**複数人になったら**: 層ごとの担当ファイルが分かれているうちは master 直でも衝突しにくいが、大きめの変更・他人の担当ファイルを触る時は `feature/xxx` ブランチ → PR → 相互レビュー。
- コミット: **小さく・検証済みで・理由を本文に**（このリポジトリの履歴が見本。何を検証したかまで書く）。
- 担当ファイルの目安（衝突防止）:
  - ① UI: `ui/app.html`
  - ② 配管: `src/server.py` `src/flight_service.py` `src/geo_safety.py`
  - ③ 守る: `safety/` ＋ ArduPilot側の差分
  - 共有: `docs/`（誰でも更新可。ただし design-decisions は決定後に）
- 提出形式（URL共有/PR）は講師確認待ち → setup-progress.md の文面案を使う。

### 公開/非公開の分割ルール（D23）

本リポジトリは**公開の課題提出用**。一度 push した内容は削除しても履歴に残るため:

- **公開リポに書かない**: 個人名・連絡先・日程・議事録（誰が何を言ったか）
- **公開リポに残す**: 設計・検証・プロセスの証跡（評価の材料。名前なしで書く）
- 担当者名入りの役割分担・MTG議事録は **private リポ**で管理:
  https://github.com/kenjiked/mimawari-team-notes （チームメンバーを collaborator に追加して共有）
- MTGの決定事項だけを、名前を伏せて design-decisions.md へD番号で転記する

## 3. MTGの回し方（60分の型）

| 時間 | 内容 |
|---|---|
| 0-5分 | 目的とゴール確認 |
| 5-15分 | **動くデモ**を全員で見る（demo-script.md）— 議論は必ず「動くもの」を土台に |
| 15-35分 | 議論（論点は事前に team-mtg-agenda.md へ） |
| 35-50分 | 決定 → その場で design-decisions.md に**D番号で追記** |
| 50-60分 | 宿題の割当（役割×「最初の一歩」= role-assignment-draft.md） |

- 決定に迷ったら判断軸は2つ: **チームが学びたいこと**と**完成可能性**（D13）。
- 議論が発散したら「3階建てのどの階の話か」を確認して仕分ける。

## 4. 週次リズム（提案）

- 週1回 MTG（上記の型）＋ 平日は非同期（コミット＋一言共有）
- 毎MTGの固定確認: ①デモが今日も完走するか ②sitl-test-log に新しい検証があるか ③締切までのスコープは守れているか

## 5. ドキュメント地図（どこに何があるか）

| 知りたいこと | 場所 |
|---|---|
| 製品は何か・決定の経緯（D1〜D22） | `design-decisions.md` ※まずこれ |
| 安全設計・しきい値の根拠・実装ロードマップ | `safety-design.md` |
| 何をどう検証したか（バグ発見含む） | `sitl-test-log.md` |
| デモの流し方 | `demo-script.md` |
| 発表原稿 / スライド構成 | `speaker-notes.md` / `presentation-outline.md` |
| 役割分担・ボトルネック | `role-assignment-draft.md` |
| MTGの進行 | `team-mtg-agenda.md` |
| チームの回し方（本書） | `team-process.md` |
| 環境構築・提出準備 | `setup-progress.md` |
| UI設計の原型 | `ux-flow.md` / `ui/mockup.html` |
| 機体側安全機構の使い方 | `safety/README.md` |

## 6. 新メンバーのオンボーディング（30分）

1. `design-decisions.md` の冒頭〜3階建てモデルを読む（10分）
2. README の手順でサーバ起動 → ブラウザでデモを一周（15分）
3. 自分の担当階を選び、`role-assignment-draft.md` の「最初の一歩」に着手（5分でタスク確認）
